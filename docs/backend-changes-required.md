# Backend Changes Required for Phase Client v2

This document describes all backend API changes needed to support the Phase Client's
core cryptographic protocol.

---

## 1. Salt Storage

The PBKDF2 salt used for key derivation must be stored per-user.

```sql
ALTER TABLE users ADD COLUMN salt TEXT NOT NULL DEFAULT '';
```

The salt is a 64-character hex string (32 random bytes) generated by the client
at registration time.

---

## 2. New Endpoint: `GET /api/v1/auth/salt`

Returns the PBKDF2 salt for a given email address so the client can derive keys
before attempting to authenticate.

```
GET /api/v1/auth/salt?email=user@example.com
```

**Response (200 OK):**
```json
{ "salt": "a3f0c9...64hexchars" }
```

**Security notes:**
- This is a public endpoint (no authentication required).
- If the email does not exist, return `404` **or** a deterministic dummy salt
  (to avoid account enumeration). The dummy salt approach is preferred:
  derive it from `HMAC(secret_server_key, email)` so the same dummy is always
  returned for the same unknown email.

---

## 3. `POST /api/v1/auth/register` — New `salt` field

The registration request body must now include the client-generated salt.

**Updated request body:**
```json
{
  "email": "user@example.com",
  "authHash": "<base64-encoded 32-byte auth hash>",
  "salt": "<64-char hex string>",
  "encryptedVault": "<base64-encoded AES-256-GCM blob>",
  "deviceName": "My Desktop"
}
```

Store `salt` alongside the user record.

---

## 4. `/health` — New `instanceSalt` field (self-hosted only)

For self-hosted deployments, the `/health` response must include a stable
`instanceSalt` so the single-user client can derive encryption keys.

**Updated response:**
```json
{
  "status": "ok",
  "initialized": true,
  "version": "1.0.0",
  "instanceSalt": "<64-char hex string>"
}
```

**Generation:** On first startup, generate 32 random bytes and persist them to
your KV store / database under a fixed key (e.g. `instance_salt`). Return the
same value on every subsequent request. Never regenerate it — doing so would
invalidate all existing encrypted vaults.

---

## 5. Self-Hosted Auth Endpoints

These endpoints are only present on self-hosted deployments and require the
`X-Phase-Instance-Token` header (see §6).

### `POST /api/v1/auth/init`

First-time vault creation. Called when `initialized: false`.

**Request:**
```json
{
  "authHash": "<base64 32 bytes>",
  "encryptedVault": "<base64 AES-256-GCM blob>"
}
```

**Response (201):**
```json
{ "token": "<jwt>" }
```

After storing the auth hash and encrypted vault, set `initialized: true` on the
instance so subsequent health checks return `initialized: true`.

### `POST /api/v1/auth/unlock`

Subsequent logins. Called when `initialized: true`.

**Request:**
```json
{
  "authHash": "<base64 32 bytes>",
  "deviceName": "My Desktop"
}
```

**Response (200):**
```json
{ "token": "<jwt>" }
```

Verify `authHash` matches stored hash (constant-time comparison), then issue JWT.

---

## 6. Instance Token Middleware

All routes on self-hosted deployments must verify the `X-Phase-Instance-Token`
header before any other processing.

```
X-Phase-Instance-Token: <token set at deploy time via INSTANCE_TOKEN env var>
```

**Behavior:**
- Missing or wrong token → `403 Forbidden` (before any auth logic runs)
- Correct token → proceed normally

This prevents port scanners from discovering or reaching login/vault endpoints.

The cloud deployment does NOT need this middleware.

## 7. Vault Endpoints (if not already implemented)

### `GET /api/v1/vault`
Returns the current encrypted vault for the authenticated user.

**Response:**
```json
{
  "encryptedVault": "<base64>",
  "version": 42
}
```

### `PUT /api/v1/vault`
Replaces the vault with optimistic concurrency control.

**Request:**
```json
{
  "encryptedVault": "<base64>",
  "expectedVersion": 42
}
```

**Response (200):**
```json
{ "version": 43 }
```

**Conflict (409):** Return error if `expectedVersion` does not match the current
server version.

---

## 8. Session Management (if not already implemented)

### `GET /api/v1/auth/sessions`
Returns all active sessions for the authenticated user.

**Response:**
```json
{
  "sessions": [
    {
      "id": "uuid",
      "deviceName": "My Desktop",
      "createdAt": "2025-01-15T10:00:00Z",
      "lastUsed": "2025-01-15T12:30:00Z",
      "isCurrent": true
    }
  ]
}
```

### `DELETE /api/v1/auth/sessions/:id`
Revokes a specific session (invalidates its JWT).

**Response:** `204 No Content`

### `POST /api/v1/auth/logout`
Invalidates the current session's JWT.

**Response:** `204 No Content`

---

## 9. Change Password (if not already implemented)

### `POST /api/v1/auth/change-password`

**Request:**
```json
{
  "currentAuthHash": "<base64>",
  "newAuthHash": "<base64>",
  "encryptedVault": "<base64 — re-encrypted with new key>",
  "version": 42
}
```

**Response:** `204 No Content`

Verify `currentAuthHash` before applying changes.
